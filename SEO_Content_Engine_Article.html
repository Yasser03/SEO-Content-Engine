The Problem We're Solving

Creating SEO-optimized content at scale is hard. You need to:

  • Research keyword opportunities to avoid duplication
  • Write content that's genuinely useful (not AI drivel)
  • Ensure quality meets real standards (not just pass fake checks)
  • Learn from what works and improve next time

Most "content automation" tools skip steps 3 and 4. They spit out articles without caring if they're good. And they never learn.

What if your system could do all four? And improve with every article?

Stage 1: Research

An LLM picks the best next article topic based on your seed keywords and niche. Crucially, it checks the knowledge store: "Have we already covered this topic?" If yes, it picks something different.

Example: For a "startup tools" blog, Research might pick "AI productivity tools for startups" on loop 1, then "no-code automation platforms" on loop 2, ensuring no duplication.

Stage 2: Generate

The LLM writes a complete SEO-optimized article: title, meta description, structured markdown with H2/H3 sections, internal link suggestions.

BUT — all metrics are calculated deterministically (no LLM involved):

  • Word count — counted character by character
  • Keyword density — exact occurrence ratio
  • Slug generation — deterministic from title
  • Section extraction — H2 headings parsed exactly

This is intentional. Metrics must be objective and testable.

Stage 2b: Quality Gate (The Real Innovation)

Here's where most systems fail. They have no real gate. They publish everything.

Not this one. The quality gate enforces 6 deterministic rules:

Check Rule Severity
Word Count Must be 600–2500 words Hard Fail
Required Sections Must have Introduction + Conclusion Hard Fail
Keyword Density Must be 0.5–3.0% (no stuffing) Hard Fail
Meta Description Should be 120–160 chars Warning
Title Should be 30–70 chars Warning
Internal Links Should have at least 1 Warning

A composite score is calculated. If it falls below the threshold (default 0.65/1.0), the post doesn't publish. Period.

✓ This is real. We've tested it. Posts that violate these rules are actually blocked. The system learns why they failed and tries to avoid the same mistakes next time.

Stage 3: Publish

Only posts that pass the quality gate reach here. The article is saved as a markdown file with YAML frontmatter (title, keywords, quality score, etc.).

You can also configure a webhook endpoint to send to Contentful, Ghost, or any CMS API.

Stage 4: Evaluate

Now the published article is scored on 5 dimensions:

  • Semantic Coverage (LLM) — Does it actually cover the topic thoroughly?
  • Keyword Strategy (deterministic) — Placement in intro/conclusion, secondary coverage
  • Readability (deterministic) — Sentence length, paragraph density, passive voice
  • Structure (deterministic) — H3 subheadings, lists, visual scanability
  • Internal Linking (deterministic) — Number and relevance of internal links

Each dimension is weighted and combined into an overall score (0.0–1.0).

Stage 5: Learn

This is the feedback loop. All evaluation results are written to a persistent JSON knowledge base:

  • ✓ Topic is marked as covered (prevents loop 2 from picking it again)
  • ✓ High-scoring structure is recorded (loop 2 will try to emulate it)
  • ✓ Feedback is logged (loop 2 will try to fix identified problems)
  • ✓ Quality trends are tracked (rolling averages, best/worst scores)
The magic: Loop 2 literally reads the knowledge store at startup. It sees: "Loop 1 tried structure X and scored 0.804. Use that." It sees: "Loop 1 forgot to cover secondary keyword Y. Don't do that." This is real learning.

Why This Design Matters

1. Explicit State Flow (No Hidden Mutation)

Each stage receives a PipelineState object, populates its output field, and returns the state. No globals. No hidden state changes. This means:

  • Easy to debug (full history visible)
  • Easy to test (deterministic inputs → deterministic outputs)
  • Easy to extend (add new stages without breaking existing ones)

2. Deterministic Quality Gate (Actually Real)

Most "AI content" systems have fake quality gates. They claim to check things but they don't. Our quality gate is:

  • Deterministic: Same input always gives same result. No randomness.
  • Testable: We have 31 tests proving it works. All passing.
  • Transparent: You know exactly why a post passed or failed.
  • Real: Posts that fail actually don't publish.

3. Persistent Learning (Actual Feedback Loop)

The knowledge store is the feedback mechanism. It's a simple JSON file, but:

  • Read at the start of each loop (informs Research and Generate)
  • Written at the end of each loop (from Learn stage)
  • Human-readable (you can inspect it, understand what's happening)
  • Git-trackable (version control your learning)

4. LLM vs Deterministic (Right Tool for Each Job)

Not everything needs an LLM. We use them for:

  • Topic selection (requires reasoning)
  • Article generation (requires creativity)
  • Semantic coverage assessment (requires language understanding)

Everything else is deterministic:

  • Word counting (math)
  • Keyword density (math)
  • Section extraction (parsing)
  • Readability metrics (rules)

This keeps the system fast, cheap, and testable.

Common Questions

Q: What if the quality gate fails?

A: The post doesn't publish. But the Learn stage still runs. The system captures why it failed and uses that knowledge for the next loop. This is how it improves.

Q: Can I use this with my CMS?

A: Yes. Set publish.destination to "webhook" and provide your CMS endpoint. The system will POST the article JSON to your endpoint. Tested with REST APIs.

Q: What LLM does it use?

A: Default is Groq (llama-3.3-70b, fast and free). But swap it in config: OpenAI (GPT-4o), Anthropic (Claude), or any provider you want. The abstraction handles it.

Q: How much does it cost?

A: Groq free tier is unlimited during trial. One loop costs ~3,000–5,000 tokens (~$0.002 on paid tier). So roughly $0.10 per article at scale.

Q: Can I use this for multiple clients?

A: Yes. Create separate config files: config/client_a.yaml, config/client_b.yaml. Run with: `python pipeline.py --config config/client_a.yaml --store store/client_a.json`

Q: Is the learning real or simulated?

A: It's real. We've tested it. Loop 2 literally reads Loop 1's knowledge store. It knows which topics to avoid, which structures work, what needs fixing. Not simulation.
